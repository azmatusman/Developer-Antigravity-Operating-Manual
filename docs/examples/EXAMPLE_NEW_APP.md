# Example: Build a New Application (End-to-End)

This example shows how to use your setup to build **any new application** with high quality and low token waste.

## Scenario (generic)
Build a small service with:
- a domain model
- persistence interface
- a simple API interface
- tests

> The point is the **process**, not the tech stack.

---

## Step 0 — Frame the task
Write 2–6 sentences:
- What problem is solved
- What is in scope
- What is out of scope
- Constraints

## Step 1 — Brainstorm (only if unclear)
Use brainstorming only if requirements are ambiguous.
Output: success criteria + boundaries.

## Step 2 — /plan (architecture + tasks)
Ask for:
- 2–3 architectures with tradeoffs
- one recommendation
- `implementation_plan.md` + `task_checklist.md`
Stop for approval.

## Step 3 — Implement using /tdd
Take one checklist item at a time.
- Write one failing test
- Implement minimum
- Refactor

## Step 4 — /review gate
Generate issue list and verdict.

## Step 5 — /document
Update README or onboarding if the setup changed.

## Step 6 — /close-session
Write SUMMARY.md and create KI only if justified.

---

## What makes this token-efficient?
- You did not re-explain process each time (workflows)
- You used the right skill for the job (skills)
- You used the right model tier for the task (model selection)
- You avoided rereading the repo (Graphify)
- You avoided hallucinated APIs (Context7)
